# 公众号发布 Skill

> 用户提供任意内容（完整文章 / 聊天记录 / 零散想法）→ Agent 改写为公众号风格文章 → 自动加 :::block 增强排版 → 生成封面 → 推送微信草稿箱

## 核心流程

### Agent 完整执行步骤

```
步骤0：判断输入类型（完整文章 / 聊天记录 / 零散想法）
步骤1：根据需要进行风格改写（聊天记录和零散想法必须改写）
步骤2：分析文章结构，自动插入 :::block 排版模块
步骤3：检查文章  →  wechat-pub inspect article.md --json
步骤4：生成封面   →  wechat-pub generate-cover --article article.md --format png --json
步骤5：转换+推送  →  wechat-pub publish article.md --auto-cover --json
```

### 步骤0：判断输入类型

| 输入类型 | 特征 | 处理方式 |
|---------|------|---------|
| 完整文章 | 有标题、段落分明、结构完整 | 跳过步骤1，直接进入步骤2 |
| 聊天记录 | 多轮对话、口语化、重复表达 | 进入步骤1改写 |
| 零散想法 | 只有要点或片段、缺结构 | 进入步骤1改写 |

**判断规则**：如果内容包含多轮对话（Q&A 交替）、大量口语化表达（"不是"、"我的意思是"、"就是"）、或缺少完整段落结构，判定为需要改写。

### 步骤1：风格改写

将零散内容整理为结构化的公众号文章。

#### 改写规则

1. **结构重组**：按「问题/场景引入 → 核心内容 → 总结/行动」三段式组织
2. **去口语化**：去掉"不是"、"我的意思是"、"就是"等口语冗余，保留核心信息
3. **不改变原意**：改写是整理和润色，不是重写，保留原作者的观点和立场
4. **适度展开**：对过于简略的要点可以补充一两句解释，但不加虚构内容
5. **保留专业术语**：如果原内容涉及专业概念，保留术语但加通俗解释

#### 语言风格规范

**核心原则：真实、自然、不端着**

##### 整体风格

- **专业但不晦涩**：涉及专业问题时保持准确性，但用生活化的类比帮助理解
- **通俗但不浅薄**：用日常语言解释复杂概念，不等于降低内容质量
- **真人口吻**：像跟懂行的朋友聊天，不像教科书，也不像营销号
- **有温度**：适当体现思考过程和真实感受，不做无感情的陈述机器

##### 禁止的 AI 味表达

以下表达方式是 AI 生成的典型特征，**必须避免**：

- ❌ "不是 A，而是 B" 句式（如"不是技术问题，而是思维问题"）
- ❌ "真正的 X 是 Y"（如"真正的挑战不是技术，而是思维"）
- ❌ "事实上"/"实际上" 开头的转折
- ❌ "值得注意的是"/"需要指出的是"
- ❌ "让我们..."（如"让我们来看看"）
- ❌ "在...的世界里"/"在...的时代"
- ❌ "这不仅是...更是..."
- ❌ 排比三连（三个结构相同的短句排在一起）
- ❌ "正如我们所见"/"正如前文所述"
- ❌ 每段结尾都来一句总结升华

##### 推荐的表达方式

- ✅ 直接说结论，不用铺垫句
- ✅ 用"比如"引出具体例子，代替抽象论述
- ✅ 用短句，避免一个句子套三个从句
- ✅ 承认不确定性（"我不确定，但我的理解是..."）比假装笃定更真实
- ✅ 技术内容先给结论，再解释原因
- ✅ 复杂概念用类比，一次讲不通就换个角度再讲

##### 不同内容类型的风格适配

| 内容类型 | 语气 | 示例 |
|---------|------|------|
| 技术工具 | 实操导向，直给方案 | "装完跑了一下，还行。" |
| 法律知识 | 专业严谨，但用案例讲 | "这个案例里，业主输在哪？" |
| 生活想法 | 轻松随意，像发朋友圈 | "最近在折腾一个东西..." |
| 知识科普 | 好奇心驱动，层层递进 | "你有没有想过，为什么..." |

### 步骤2：自动添加 :::block 排版模块

Agent 读取用户 Markdown 后，根据内容结构自动插入排版模块。

#### 插入规则

| 条件 | 插入模块 | 位置 |
|------|---------|------|
| 文章有标题/H1 | `:::hero` | 文章最开头（frontmatter 之后） |
| 文章有小标题段落，且有核心观点 | `:::verdict` | 核心观点段落处 |
| 文章有提示/注意/警告性质的内容 | `:::callout` | 提示内容处 |
| 文章有3个以上小节 | `:::toc` | hero 之后、正文之前 |
| 文章末尾需要行动号召 | `:::cta` | 文章末尾 |
| 文章有问答内容 | `:::faq` | 问答段落处 |

#### 排版模块语法（三个冒号）

```markdown
:::hero
eyebrow: 简短分类标签
title: 文章主标题
subtitle: 一句话副标题
:::

:::callout
type: tip
body: 提示内容文字
:::

:::verdict
eyebrow: 标签
title: 核心判断句
body: 展开说明
:::

:::toc
items:
  - 目录项1
  - 目录项2
:::

:::faq
items:
  - question: 问题？
    answer: 回答。
:::

:::cta
title: 行动号召标题
button_text: 按钮文字
button_url: https://example.com
:::
```

> 可省略 `block` 关键字：`:::hero` 等同于 `:::block hero`

#### 示例：纯 Markdown → 增强后

**用户写的原始 Markdown：**
```markdown
## 为什么你的文章读者读不完

你有没有花了一整天写的好文章，发出去后阅读量惨淡？

问题可能不是内容，而是排版。好的排版要在 3 秒内让读者决定继续读。

## 排版的核心

排版的本质不是审美，而是降低读者的决策成本。每一次在手机上阅读，读者都在做决策：继续还是划走。

## 开始行动

从今天开始重构你的公众号文章吧。
```

**Agent 自动增强后：**
```markdown
:::hero
eyebrow: 排版思维
title: 为什么你的文章读者读不完
subtitle: 手机上3秒没给读者理由继续
:::

你有没有花了一整天写的好文章，发出去后阅读量惨淡？

:::callout
type: tip
body: 排版不是让文章"好看"，而是让读者更容易做出"继续读"的决定。
:::

## 排版的核心

:::verdict
eyebrow: 核心理念
title: 排版的本质是降低读者的决策成本
body: 每一次在手机上阅读，读者都在做决策：继续还是划走。排版要做的是让"继续"这个决策更容易。
:::

## 开始行动

:::cta
title: 从今天开始，重构你的公众号文章
button_text: 立即开始
button_url: #
:::
```

#### 模块使用原则

- **不超过 5 个**：一篇文章 hero 1个 + verdict 1个 + cta 1个 足够，callout 0-2个
- **hero 必有**：每篇文章开头都需要一个 hero
- **cta 必有**：文章末尾引导行动
- **verdict 最多 1 个**：一篇文章只有一个核心判断
- **不重复**：同类型模块不重复出现

## 安装

```bash
pip install md2wechat-py
```

或从源码安装：

```bash
git clone https://github.com/zh2673-git/md2wechat-py.git
cd md2wechat-py
pip install -e .
```

## 使用前必做

```bash
wechat-pub capabilities --json    # 确认 CLI 可用
wechat-pub config init             # 配置微信凭证
```

## CLI 命令速查

| 命令 | 用途 |
|------|------|
| `wechat-pub inspect article.md --json` | 检查文章准备状态 |
| `wechat-pub convert article.md -o output.html` | 仅转换 HTML |
| `wechat-pub preview article.md` | 本地预览 |
| `wechat-pub generate-cover --article article.md --format png --json` | 生成封面 |
| `wechat-pub publish article.md --auto-cover --json` | 一键排版+封面+推送草稿 |
| `wechat-pub upload-image cover.png --type thumb --json` | 上传图片到微信素材库 |
| `wechat-pub layout list --json` | 查看排版模块列表 |
| `wechat-pub layout show hero --json` | 查看某个模块详情 |
| `wechat-pub config init` | 初始化微信凭证配置 |
| `wechat-pub config show` | 查看当前配置 |
| `wechat-pub capabilities --json` | 查看 CLI 能力 |

## Agent 规则

1. **先判断输入类型**：完整文章跳过改写，聊天记录/零散想法先改写
2. **⚠️ 字数优化约束（关键）**：
    - 微信公众号文章内容上限为 **20000 字符**（含 HTML 标签）
    - **目标字数：19000-20000 字符** —— 在限制内尽可能充实内容
    - 改写时**充分展开**：补充详细解释、增加案例、类比、具体细节和深度分析
    - **禁止过度精简**：不要为了"简洁"而删减有价值的内容
    - **避免堆砌源码**：涉及项目代码时，不要大段列举源码，用精简示例说明核心逻辑即可
    - 检查方式：转换后的 HTML 大小可通过 publish 命令输出查看
3. **风格改写优先**：改写是"整理+润色"，不是"重写"，保留原意和原作者立场
4. **自动增强排版**：用户只需提供纯 Markdown，Agent 负责分析内容并自动插入 :::block 排版模块
5. **先 inspect 再操作**：增强完 :::block 后，先 inspect --json 确认无 error
6. **先 capabilities**：不确定命令是否存在时，先执行 capabilities --json
7. **模块不超过 5 个**：hero + verdict + cta 是标配，callout 按需
8. **草稿需要封面**：没有封面时用 --auto-cover 自动生成
9. **摘要自动生成**：不需要手动指定 digest，微信会自动截取
10. **JSON 输出**：Agent 调用 CLI 时始终加 --json，便于解析
11. **确认后再推送**：先 preview 让用户确认效果，再 publish
12. **微信凭证安全**：配置文件在 ~/.config/wechat-pub/config.yaml，不在项目目录内，不会随代码提交
13. **避免 AI 味**：严格遵循语言风格规范中的禁用表达列表

## 主题

| 主题名 | 风格 |
|--------|------|
| `claude-warm` | 暖阳：赭石橙+奶油白（默认，推荐） |
| `claude-clean` | 极简：白底+赭石点缀 |
| `claude-dark` | 暗夜：深色暖棕调 |

## 关于封面

- `generate-cover` 生成 Claude 风格封面（标题+几何装饰+主题色）
- 如需 AI 配图，Agent 可用 image_gen 工具生成，然后 `publish --cover 图片.png` 指定

## 微信公众号配置

1. 登录 [微信公众平台](https://mp.weixin.qq.com)
2. 进入「开发 → 基本配置」→ 获取 AppID 和 AppSecret
3. 将服务器 IP 加入白名单（或使用测试号）
4. 运行 `wechat-pub config init`，配置文件在 `~/.config/wechat-pub/config.yaml`
5. 或设置环境变量：`WECHAT_APP_ID` / `WECHAT_APP_SECRET`

**安全说明**：配置文件存储在用户主目录 `~/.config/wechat-pub/` 下，不在项目目录内，推送到 GitHub 不会暴露凭证。

**测试号**：如需测试，可在「开发 → 开发者工具」申请测试号，不受 IP 白名单限制。

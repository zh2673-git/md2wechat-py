# 公众号发布 Skill

> 用户提供任意内容（完整文章 / 聊天记录 / 零散想法 / 笔记文件）→ Agent 判断类型并改写 → 自动加 :::block 增强排版 → 生成封面 → 推送微信草稿箱

## 核心流程

### Agent 完整执行步骤

```
步骤0：扫描输入源，确定文章分组
步骤1：对每个分组分别处理——判断输入类型（完整文章 / AI聊天记录 / 笔记 / 零散想法）
步骤2：根据类型决定输出策略：
       - AI聊天记录 → 生成两个版本（对话整理版 + 论文风格版）
       - 笔记/零散想法 → 生成一个版本（论文风格版）
       - 完整文章 → 直接进入排版（跳过改写）
步骤3：对每个版本分别进行风格改写
步骤4：分析文章结构，自动插入 :::block 排版模块
步骤5：检查文章  →  wechat-pub inspect article.md --json
步骤6：生成封面   →  wechat-pub generate-cover --article article.md --format png --json
步骤7：转换+推送  →  wechat-pub publish article.md --auto-cover --json
```

### 步骤0：扫描输入源，确定文章分组

Agent 支持三种输入方式，按优先级处理：

#### 方式1：`data/` 文件夹批量模式（推荐）

用户在当前目录下创建 `data/` 文件夹，Agent 自动扫描并确定文章分组：

```
data/
├── 话题A/              ← 子文件夹 → 合并为1篇文章
│   ├── 01_intro.md
│   ├── 02_detail.md
│   └── 03_summary.md
├── 话题B/              ← 子文件夹 → 合并为1篇文章
│   ├── chat_with_ai.md
│   └── notes.md
├── 独立笔记1.md        ← 直接的 .md 文件 → 各生成1篇文章
└── 独立笔记2.md        ← 直接的 .md 文件 → 各生成1篇文章
```

**分组规则**：
- `data/` 下的**子文件夹** → 文件夹内所有 .md 文件合并为1篇文章
- `data/` 下的**直接 .md 文件** → 每个文件各生成1篇文章
- 忽略非 .md 文件

**子文件夹内合并规则**：
- 按文件名自然排序（01_ → 02_ → 03_）
- 每个文件之间用 `---` 分隔
- 合并后统一进入步骤1判断类型

#### 方式2：用户直接指定文件

用户直接告诉 Agent 要处理哪些文件，Agent 按「步骤0：收集与合并输入（旧版）」处理。

#### 方式3：用户提供文本内容

用户直接在对话中粘贴内容，Agent 按内联内容处理。

#### Agent 自动检测逻辑

1. 检查当前目录下是否存在 `data/` 文件夹
2. 如果存在 → 扫描 `data/`，按分组规则确定文章列表
3. 如果不存在 → 询问用户要处理哪些文件，或等待用户指定

#### 批量处理输出

当一次处理多篇文章时，Agent 逐篇执行步骤1-7，每篇独立生成和推送。生成的改写稿（对话版/论文版）统一保存在 `output/` 文件夹下，以 `{话题名}_对话版.md` 和 `{话题名}_论文版.md` 命名。输出汇总：

```
处理完成：
  ✅ 话题A → 论文风格版（已推送）  → output/话题A_论文版.md
  ✅ 话题B → 对话整理版 + 论文风格版（已推送2篇） → output/话题B_对话版.md + output/话题B_论文版.md
  ✅ 独立笔记1.md → 论文风格版（已推送）  → output/独立笔记1_论文版.md
  ✅ 独立笔记2.md → 论文风格版（已推送）  → output/独立笔记2_论文版.md
```

### 步骤1：判断输入类型

| 输入类型 | 特征 | 输出策略 |
|---------|------|---------|
| 完整文章 | 有标题、段落分明、结构完整 | 单输出，跳过改写 |
| AI聊天记录 | 多轮人机对话、有"你"/"我"交替、口语化 | **双输出**：对话整理版 + 论文风格版 |
| 笔记文件 | 有结构化标题和要点、非对话形式 | 单输出，论文风格版 |
| 零散想法 | 只有要点或片段、缺结构 | 单输出，论文风格版 |

**AI聊天记录判断规则**（满足任意2条即判定）：
- 包含明确的角色标记（如"用户:"、"AI:"、"Human:"、"Assistant:"、"你:"、"我:" 交替出现）
- 存在多轮问答结构（问题→回答→追问→回答）
- 包含大量口语化表达（"不是"、"我的意思是"、"就是"、"对对对"）
- 内容中出现典型的AI交互痕迹（"好的，让我来"、"作为AI"）

**笔记文件判断规则**：
- 有结构化标题（H1/H2/H3）
- 内容以知识点、要点、摘要为主
- 无对话交替结构

### 步骤2：输出策略详解

#### 策略A：AI聊天记录 → 双输出

当检测到输入为AI聊天记录时，Agent 生成 **两个独立的公众号草稿**：

**输出1：对话整理版**
- 保留对话的 Q&A 结构，但经过整理和润色
- 去掉口语冗余、重复表达、无意义的寒暄
- 保留有价值的追问和深入讨论
- 语气风格：轻松但有条理，像播客的文字版

**输出2：论文风格版**
- 将对话内容完全重组为论文/深度文章结构
- 提取核心论点，按逻辑链组织
- 加入引用标注（如果原文有数据支撑）
- 语气风格：专业严谨，但不晦涩

**两个版本的标题区分**：
- 对话整理版：标题后缀 `（对话版）` 或标题本身体现对话性质
- 论文风格版：独立标题，体现论点

#### 策略B：笔记/零散想法 → 单输出（论文风格版）

- 将笔记内容重组为论文/深度文章结构
- 提取核心主题，按逻辑组织
- 适度展开简略的要点
- 语气风格：专业严谨，但不晦涩

#### 策略C：完整文章 → 单输出（跳过改写）

- 直接进入步骤3排版增强

### 步骤3：风格改写

#### 3a：对话整理版改写规则（仅AI聊天记录）

1. **保留对话骨架**：保持 Q&A 结构，问题作为引子，回答作为主体
2. **去口语化**：去掉"不是"、"我的意思是"、"就是"等口语冗余
3. **精简寒暄**：去掉"你好"、"好的"、"没问题"等无意义交互
4. **合并相关问答**：同一话题的多次追问可以合并为一个完整段落
5. **保留追问深度**：有价值的追问和深入讨论要保留，这是对话版的独特价值
6. **标注角色**：提问者标注为 **「我：」**，回答者标注为 **「AI：」**。如果原文有其他角色（如第三人转述），保留原角色名标注
7. **角色不可混淆**：「我」始终是原文的人类提问者，「AI」始终是回答者，不可互换

#### 3b：论文风格版改写规则（AI聊天记录 + 笔记 + 零散想法）

1. **结构重组**：按「引言/背景 → 核心论点（2-4个）→ 深度分析 → 结论/展望」四段式组织
2. **提炼论点**：从对话/笔记中提取2-4个核心论点，每个论点独立成节
3. **去口语化**：去掉所有口语表达，用书面语替代
4. **补充论证**：对过于简略的论点补充论证逻辑，但不加虚构内容
5. **保留专业术语**：专业概念保留术语但加通俗解释
6. **引用溯源**：如果原文中有数据、案例，在文中注明来源
7. **逻辑衔接**：段落之间用逻辑连接词过渡，避免突兀跳跃

#### 语言风格规范

**核心原则：真实、自然、不端着**

> 两种输出版本有不同的风格侧重，请根据版本类型选择合适的风格。

##### 整体风格（对话整理版）

- **对谈感**：保留"提问-回答"的节奏感，像播客的文字版
- **自然流畅**：去掉了口语冗余，但仍然是"说话"的方式
- **有温度**：适当保留思考过程中的犹豫和修正，这是对话版的独特魅力
- **节奏明快**：一问一答推进，不拖沓

##### 整体风格（论文风格版）

- **专业但不晦涩**：涉及专业问题时保持准确性，但用生活化的类比帮助理解
- **逻辑严密**：论点之间有清晰的逻辑链条，不跳跃
- **客观理性**：像一篇好的专栏文章，有立场但不情绪化
- **深度优先**：宁可少讲一个点，也要把讲到的点说透

##### 禁止的 AI 味表达

以下表达方式是 AI 生成的典型特征，**必须避免**：

- ❌ "不是 A，而是 B" 句式（如"不是技术问题，而是思维问题"）
- ❌ "真正的 X 是 Y"（如"真正的挑战不是技术，而是思维"）
- ❌ "事实上"/"实际上" 开头的转折
- ❌ "值得注意的是"/"需要指出的是"
- ❌ "让我们..."（如"让我们来看看"）
- ❌ "在...的世界里"/"在...的时代"
- ❌ "这不仅是...更是..."
- ❌ 排比三连（三个结构相同的短句排在一起）
- ❌ "正如我们所见"/"正如前文所述"
- ❌ 每段结尾都来一句总结升华

##### 推荐的表达方式

- ✅ 直接说结论，不用铺垫句
- ✅ 用"比如"引出具体例子，代替抽象论述
- ✅ 用短句，避免一个句子套三个从句
- ✅ 承认不确定性（"我不确定，但我的理解是..."）比假装笃定更真实
- ✅ 技术内容先给结论，再解释原因
- ✅ 复杂概念用类比，一次讲不通就换个角度再讲

##### 不同内容类型的风格适配

| 输出类型 | 语气 | 示例 |
|---------|------|------|
| 对话整理版 | 对谈感，像播客文字版 | "Q：这个问题怎么解决？A：其实核心在于..." |
| 论文风格版-技术工具 | 实操导向，直给方案 | "装完跑了一下，还行。" |
| 论文风格版-法律知识 | 专业严谨，但用案例讲 | "这个案例里，业主输在哪？" |
| 论文风格版-生活想法 | 轻松随意，像发朋友圈 | "最近在折腾一个东西..." |
| 论文风格版-知识科普 | 好奇心驱动，层层递进 | "你有没有想过，为什么..." |

### 步骤4：自动添加 :::block 排版模块

> **核心原则：内容第一，样式第二。** 排版模块是为了提升阅读体验，而不是为了填塞显示效果。该用就用，不该用就不用，以"有实际价值"为判断标准。

Agent 读取用户 Markdown 后，根据内容结构**按需**插入排版模块。

#### 插入规则

| 条件 | 推荐插入 | 位置 | 说明 |
|------|---------|------|------|
| 所有文章 | `:::hero` | 文章最开头 | **必选**，取代传统标题，增强首页预览效果 |
| 文章末尾 | `:::cta` | 文章末尾 | **必选**，引导关注或互动 |
| 有明确核心判断 | `:::verdict` | 核心观点处 | 推荐，1个即可，突出文章的中心思想 |
| 有需要重点提示的内容 | `:::callout` | 提示内容处 | 按需，0-2个，用于重要但非正文的信息 |
| 文章有3个以上小节 | `:::toc` | hero之后正文之前 | 按需，适合长文，帮助读者建立阅读预期 |
| 有问答结构（非对话版） | `:::faq` | 问答段落处 | 按需，适合论文版中的问答环节 |

**判断原则**：问自己"这个模块对读者有用吗？"——有用就加，只是为了好看就不加。

#### 排版模块语法（三个冒号）

```markdown
:::hero
eyebrow: 简短分类标签
title: 文章主标题
subtitle: 一句话副标题
:::

:::callout
type: tip
body: 提示内容文字
:::

:::verdict
eyebrow: 标签
title: 核心判断句
body: 展开说明
:::

:::toc
items:
  - 目录项1
  - 目录项2
:::

:::faq
items:
  - question: 问题？
    answer: 回答。
:::

:::cta
title: 行动号召标题
button_text: 按钮文字
button_url: https://example.com
:::
```

> 可省略 `block` 关键字：`:::hero` 等同于 `:::block hero`

#### 示例：纯 Markdown → 增强后

**用户写的原始 Markdown：**
```markdown
## 为什么你的文章读者读不完

你有没有花了一整天写的好文章，发出去后阅读量惨淡？

问题可能不是内容，而是排版。好的排版要在 3 秒内让读者决定继续读。

## 排版的核心

排版的本质不是审美，而是降低读者的决策成本。每一次在手机上阅读，读者都在做决策：继续还是划走。

## 开始行动

从今天开始重构你的公众号文章吧。
```

**Agent 自动增强后（极简模块，内容优先）：**
```markdown
:::hero
eyebrow: 排版思维
title: 为什么你的文章读者读不完
subtitle: 手机上3秒没给读者理由继续
:::

你有没有花了一整天写的好文章，发出去后阅读量惨淡？排版不是让文章"好看"，而是让读者更容易做出"继续读"的决定。好的排版要在 3 秒内让读者决定继续读。

## 排版的核心

:::verdict
eyebrow: 核心理念
title: 排版的本质是降低读者的决策成本
body: 每一次在手机上阅读，读者都在做决策：继续还是划走。排版要做的是让"继续"这个决策更容易。
:::

## 开始行动

从今天开始重构你的公众号文章吧。

:::cta
title: 从今天开始，重构你的公众号文章
button_text: 立即开始
button_url: #
:::
```

#### 模块使用原则

- **内容优先**：20000 字符上限里，样式字符越少，内容空间越大
- **最少模块**：一篇文章只用 hero + cta 两个模块，verdict 按需加1个，总共不超过3个
- **hero 必选**：每篇文章开头一个 hero，替代传统标题
- **cta 必选**：文章末尾引导行动
- **verdict 最多1个**：只有确实有核心判断时才加
- **不重复**：同类型模块不重复出现
- **callout/toc/faq 尽量不用**：这些模块的 body 内容往往和正文重复，浪费字符空间

## CLI 命令速查

| 命令 | 用途 |
|------|------|
| `wechat-pub inspect article.md --json` | 检查文章准备状态 |
| `wechat-pub convert article.md -o output.html` | 仅转换 HTML |
| `wechat-pub preview article.md` | 本地预览 |
| `wechat-pub generate-cover --article article.md --format png --json` | 生成封面 |
| `wechat-pub publish article.md --auto-cover --json` | 一键排版+封面+推送草稿 |
| `wechat-pub upload-image cover.png --type thumb --json` | 上传图片到微信素材库 |
| `wechat-pub layout list --json` | 查看排版模块列表 |
| `wechat-pub layout show hero --json` | 查看某个模块详情 |
| `wechat-pub config init` | 初始化微信凭证配置 |
| `wechat-pub config show` | 查看当前配置 |
| `wechat-pub capabilities --json` | 查看 CLI 能力 |

## Agent 规则

1. **优先扫描 data/ 文件夹**：检查当前目录下 `data/` 文件夹是否存在，存在则按分组规则批量处理
2. **再判断输入类型**：完整文章跳过改写，AI聊天记录双输出，笔记/零散想法单输出
3. **批量逐篇处理**：多篇文章逐篇执行步骤1-7，每篇独立改写、排版、推送，最后输出汇总
4. **⚠️ 字数优化约束（关键）**：
    - 微信公众号文章内容上限为 **20000 字符**（含 HTML 标签）
    - **目标字数：19000-20000 字符** —— 在限制内尽可能充实内容
    - **内容第一，样式第二**：20000 字符里，正文内容占大头，排版样式服务于内容而不是替代内容
    - **排版适量不过度**：hero + cta 是标配，verdict/callout/toc/faq 按需使用，对读者阅读体验有益就加，仅为好看就不加
    - **充分展开是硬约束，不是建议**：
      * 对原文中的每个关键论点，至少补充 **2-3 个具体例子、类比或延伸分析**
      * 每个回答段落展开到 200-400 字（中文），不要只有寥寥几句
      * 引入同类比较（如"这与 X 类似，但不同之处在于……"）
      * 添加"如果……会怎样"的假设性推演，拓宽讨论维度
      * 对专业概念做通俗解释 + 实际应用场景举例
    - **禁止过度精简**：不要为了"简洁"而删减有价值的内容
    - **检查方式**：转换后的 HTML 大小可通过 publish 命令输出查看，如果未达 19000 字符需继续补充
5. **双输出处理**：AI聊天记录需要生成两个独立的 Markdown 文件，分别推送两个草稿
6. **风格区分**：对话整理版和论文风格版使用不同的改写规则和语言风格，不能互相混淆
7. **自动增强排版**：用户只需提供纯 Markdown，Agent 负责分析内容并**按需**插入 :::block 排版模块
8. **先 inspect 再操作**：增强完 :::block 后，先 inspect --json 确认无 error
9. **先 capabilities**：不确定命令是否存在时，先执行 capabilities --json
10. **排版适量，内容优先**：hero + cta 是标配，verdict/callout/toc/faq 按需使用，以"对读者有用"为标准，不堆砌样式模块
11. **草稿需要封面**：没有封面时用 --auto-cover 自动生成
12. **摘要自动生成**：不需要手动指定 digest，微信会自动截取
13. **JSON 输出**：Agent 调用 CLI 时始终加 --json，便于解析
14. **确认后再推送**：先 preview 让用户确认效果，再 publish
15. **微信凭证安全**：配置文件在 ~/.config/wechat-pub/config.yaml，不在项目目录内，不会随代码提交
16. **避免 AI 味**：严格遵循语言风格规范中的禁用表达列表
17. **文件存放**：生成的改写稿统一保存在 `output/` 文件夹下，以 `{话题名}_对话版.md` 和 `{话题名}_论文版.md` 命名。`output/` 不纳入 Git 仓库
18. **data/ 文件夹不提交**：`data/` 是临时输入目录，不应出现在 git 仓库中

## 微信公众号配置

1. 登录 [微信公众平台](https://mp.weixin.qq.com)
2. 进入「开发 → 基本配置」→ 获取 AppID 和 AppSecret
3. 将服务器 IP 加入白名单（或使用测试号）
4. 运行 `wechat-pub config init`，配置文件在 `~/.config/wechat-pub/config.yaml`
5. 或设置环境变量：`WECHAT_APP_ID` / `WECHAT_APP_SECRET`

**安全说明**：配置文件存储在用户主目录 `~/.config/wechat-pub/` 下，不在项目目录内，推送到 GitHub 不会暴露凭证。

**测试号**：如需测试，可在「开发 → 开发者工具」申请测试号，不受 IP 白名单限制。

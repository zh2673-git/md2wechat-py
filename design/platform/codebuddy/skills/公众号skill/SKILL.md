# 公众号发布 Skill

> 用户提供任意内容（完整文章 / 聊天记录 / 零散想法 / 笔记文件）→ Agent 判断类型并改写 → 自动用原生 Markdown 增强排版 → 自动用 SVG/AI 生成配图并插入 → 生成封面 → 推送微信草稿箱

## 核心流程

### Agent 完整执行步骤

```
步骤0：扫描输入源，确定文章分组
步骤1：对每个分组分别处理——判断输入类型（完整文章 / AI聊天记录 / 笔记 / 零散想法）
步骤2：根据类型决定输出策略：
       - AI聊天记录 → 生成两个版本（对话整理版 + 论文风格版）
       - 笔记/零散想法 → 生成一个版本（论文风格版）
       - 完整文章 → 直接进入排版（跳过改写）
步骤3：对每个版本分别进行风格改写
步骤3.5：材料验证与引用溯源 → 网络搜索相关材料，校对引用出处的真实性和准确性
步骤4：分析文章结构，用原生 Markdown 增强排版
步骤4.5：分析文章内容，自动生成配图并插入对应位置
步骤5：检查文章  →  wechat-pub inspect article.md --json
步骤6：字数检查   →  确认纯文本字数在7000-8000字之间，总字符（含HTML和配图）在19000-20000之间
步骤7：生成封面   →  wechat-pub generate-cover --article article.md --format png --json
步骤8：转换+推送  →  wechat-pub publish article.md --auto-cover --author "公众号名"
```

### 步骤0：扫描输入源，确定文章分组

Agent 支持三种输入方式，按优先级处理：

#### 方式1：`data/` 文件夹批量模式（推荐）

用户在当前目录下创建 `data/` 文件夹，Agent 自动扫描并确定文章分组：

```
data/
├── 话题A/              ← 子文件夹 → 合并为1篇文章
│   ├── 01_intro.md
│   ├── 02_detail.md
│   └── 03_summary.md
├── 话题B/              ← 子文件夹 → 合并为1篇文章
│   ├── chat_with_ai.md
│   └── notes.md
├── 独立笔记1.md        ← 直接的 .md 文件 → 各生成1篇文章
└── 独立笔记2.md        ← 直接的 .md 文件 → 各生成1篇文章
```

**分组规则**：
- `data/` 下的**子文件夹** → 文件夹内所有 .md 文件合并为1篇文章
- `data/` 下的**直接 .md 文件** → 每个文件各生成1篇文章
- 忽略非 .md 文件

**子文件夹内合并规则**：
- 按文件名自然排序（01_ → 02_ → 03_）
- 每个文件之间用 `---` 分隔
- 合并后统一进入步骤1判断类型

#### 方式2：用户直接指定文件

用户直接告诉 Agent 要处理哪些文件，Agent 按指定文件处理。

#### 方式3：用户提供文本内容

用户直接在对话中粘贴内容，Agent 按内联内容处理。

#### Agent 自动检测逻辑

1. 检查当前目录下是否存在 `data/` 文件夹
2. 如果存在 → 扫描 `data/`，按分组规则确定文章列表
3. 如果不存在 → 询问用户要处理哪些文件，或等待用户指定

#### 批量处理输出

当一次处理多篇文章时，Agent 逐篇执行步骤1-7，每篇独立生成和推送。生成的改写稿（对话版/论文版）统一保存在 `output/` 文件夹下，以 `{话题名}_对话版.md` 和 `{话题名}_论文版.md` 命名。输出汇总：

```
处理完成：
  ✅ 话题A → 论文风格版（已推送）  → output/话题A_论文版.md
  ✅ 话题B → 对话整理版 + 论文风格版（已推送2篇） → output/话题B_对话版.md + output/话题B_论文版.md
  ✅ 独立笔记1.md → 论文风格版（已推送）  → output/独立笔记1_论文版.md
  ✅ 独立笔记2.md → 论文风格版（已推送）  → output/独立笔记2_论文版.md
```

### 步骤1：判断输入类型

| 输入类型 | 特征 | 输出策略 |
|---------|------|---------|
| 完整文章 | 有标题、段落分明、结构完整 | 单输出，跳过改写 |
| AI聊天记录 | 多轮人机对话、有"你"/"我"交替、口语化 | **双输出**：对话整理版 + 论文风格版 |
| 笔记文件 | 有结构化标题和要点、非对话形式 | 单输出，论文风格版 |
| 零散想法 | 只有要点或片段、缺结构 | 单输出，论文风格版 |

**AI聊天记录判断规则**（满足任意2条即判定）：
- 包含明确的角色标记（如"用户:"、"AI:"、"Human:"、"Assistant:"、"你:"、"我:" 交替出现）
- 存在多轮问答结构（问题→回答→追问→回答）
- 包含大量口语化表达（"不是"、"我的意思是"、"就是"、"对对对"）
- 内容中出现典型的AI交互痕迹（"好的，让我来"、"作为AI"）

**笔记文件判断规则**：
- 有结构化标题（H1/H2/H3）
- 内容以知识点、要点、摘要为主
- 无对话交替结构

### 步骤2：输出策略详解

#### 策略A：AI聊天记录 → 双输出

当检测到输入为AI聊天记录时，Agent 生成 **两个独立的公众号草稿**：

**输出1：对话整理版**
- 保留对话的 Q&A 结构，但经过整理和润色
- 去掉口语冗余、重复表达、无意义的寒暄
- 保留有价值的追问和深入讨论
- 语气风格：轻松但有条理，像播客的文字版

**输出2：论文风格版**
- 将对话内容完全重组为论文/深度文章结构
- 提取核心论点，按逻辑链组织
- 加入引用标注（如果原文有数据支撑）
- 语气风格：专业严谨，但不晦涩

**两个版本的标题区分**：
- 对话整理版：标题后缀 `（对话版）` 或标题本身体现对话性质
- 论文风格版：独立标题，体现论点

#### 策略B：笔记/零散想法 → 单输出（论文风格版）

- 将笔记内容重组为论文/深度文章结构
- 提取核心主题，按逻辑组织
- 适度展开简略的要点
- 语气风格：专业严谨，但不晦涩

#### 策略C：完整文章 → 单输出（跳过改写）

- 直接进入步骤4排版增强

### 步骤3：风格改写

#### 3a：对话整理版改写规则（仅AI聊天记录）

1. **保留对话骨架**：保持 Q&A 结构，问题作为引子，回答作为主体
2. **去口语化**：去掉"不是"、"我的意思是"、"就是"等口语冗余
3. **精简寒暄**：去掉"你好"、"好的"、"没问题"等无意义交互
4. **合并相关问答**：同一话题的多次追问可以合并为一个完整段落
5. **保留追问深度**：有价值的追问和深入讨论要保留，这是对话版的独特价值
6. **标注角色**：提问者标注为 **「我：」**，回答者标注为 **「AI：」**。如果原文有其他角色（如第三人转述），保留原角色名标注
7. **角色不可混淆**：「我」始终是原文的人类提问者，「AI」始终是回答者，不可互换

#### 3b：论文风格版改写规则（AI聊天记录 + 笔记 + 零散想法）

1. **结构重组**：按「引言/背景 → 核心论点（2-4个）→ 深度分析 → 结论/展望 → 参考文献」五段式组织
2. **提炼论点**：从对话/笔记中提取2-4个核心论点，每个论点独立成节
3. **去口语化**：去掉所有口语表达，用书面语替代
4. **补充论证**：对过于简略的论点补充论证逻辑，但不加虚构内容
5. **保留专业术语**：专业概念保留术语但加通俗解释
6. **引用溯源**：文中引用的数据、案例、法条、观点必须标注来源，来源格式为上标编号如[1]、[2]，对应文末参考文献列表
7. **逻辑衔接**：段落之间用逻辑连接词过渡，避免突兀跳跃
8. **参考文献**：文末必须附参考文献列表，列出所有引用出处的详细信息（作者、标题、来源、日期/链接），格式见步骤3.5

#### 语言风格规范

**核心原则：真实、自然、不端着**

> 两种输出版本有不同的风格侧重，请根据版本类型选择合适的风格。

##### 整体风格（对话整理版）

- **对谈感**：保留"提问-回答"的节奏感，像播客的文字版
- **自然流畅**：去掉了口语冗余，但仍然是"说话"的方式
- **有温度**：适当保留思考过程中的犹豫和修正，这是对话版的独特魅力
- **节奏明快**：一问一答推进，不拖沓

##### 整体风格（论文风格版）

- **专业但不晦涩**：涉及专业问题时保持准确性，但用生活化的类比帮助理解
- **逻辑严密**：论点之间有清晰的逻辑链条，不跳跃
- **客观理性**：像一篇好的专栏文章，有立场但不情绪化
- **深度优先**：宁可少讲一个点，也要把讲到的点说透

##### 禁止的 AI 味表达

以下表达方式是 AI 生成的典型特征，**必须避免**：

- ❌ "不是 A，而是 B" 句式（如"不是技术问题，而是思维问题"）
- ❌ "真正的 X 是 Y"（如"真正的挑战不是技术，而是思维"）
- ❌ "事实上"/"实际上" 开头的转折
- ❌ "值得注意的是"/"需要指出的是"
- ❌ "让我们..."（如"让我们来看看"）
- ❌ "在...的世界里"/"在...的时代"
- ❌ "这不仅是...更是..."
- ❌ 排比三连（三个结构相同的短句排在一起）
- ❌ "正如我们所见"/"正如前文所述"
- ❌ 每段结尾都来一句总结升华

##### 推荐的表达方式

- ✅ 直接说结论，不用铺垫句
- ✅ 用"比如"引出具体例子，代替抽象论述
- ✅ 用短句，避免一个句子套三个从句
- ✅ 承认不确定性（"我不确定，但我的理解是..."）比假装笃定更真实
- ✅ 技术内容先给结论，再解释原因
- ✅ 复杂概念用类比，一次讲不通就换个角度再讲

##### 段落与排版规范

- **段落不宜过长**：每段控制在 3-5 句话以内（约 100-200 字）。如果一段文字超过 200 字，应在逻辑转折处换行拆分为多个段落。手机屏幕窄，大段文字会让读者产生压迫感，短段落更易阅读
- **关键词适当加粗**：文章中的专业术语、核心概念、关键判断、重要数据等关键词应适当加粗（`**关键词**`），帮助读者快速扫描和定位重点。加粗的对象应是完整的概念或短语，不要只加粗单个字
- **加粗与段落配合**：每个段落最好有 1-2 处加粗关键词，既提供视觉锚点，又不破坏阅读节奏。如果一个段落没有任何加粗，考虑是否有值得强调的关键词

##### 不同内容类型的风格适配

| 输出类型 | 语气 | 示例 |
|---------|------|------|
| 对话整理版 | 对谈感，像播客文字版 | "Q：这个问题怎么解决？A：其实核心在于..." |
| 论文风格版-技术工具 | 实操导向，直给方案 | "装完跑了一下，还行。" |
| 论文风格版-法律知识 | 专业严谨，但用案例讲 | "这个案例里，业主输在哪？" |
| 论文风格版-生活想法 | 轻松随意，像发朋友圈 | "最近在折腾一个东西..." |
| 论文风格版-知识科普 | 好奇心驱动，层层递进 | "你有没有想过，为什么..." |

### 步骤3.5：材料验证与引用溯源

> **核心原则：引用必须真实，出处必须可查。** 论文风格版文章中引用的任何数据、案例、法条、观点都必须经过验证，确保真实性和准确性。不可凭记忆或推测编造引用。

Agent 在完成风格改写后，对论文风格版文章中涉及的所有外部材料执行验证流程：

#### 验证流程

```
1. 提取文章中所有涉及外部材料的陈述（数据、案例、法条、观点、统计数字等）
2. 对每条陈述，使用网络搜索工具查找原始出处
3. 校对内容准确性：
   - 数据/数字是否与原始来源一致
   - 法条引用是否准确（条款号、生效时间、适用范围）
   - 案例描述是否与公开报道一致
   - 观点归属是否正确（是否确实出自该人物/机构）
4. 验证通过的 → 在文中标注上标编号[1][2]...，文末列入参考文献
5. 验证不通过的 → 修正为准确内容或删除该引用
6. 无法验证的 → 标注"据公开报道"或删除，不列入参考文献
```

#### 参考文献格式

文末参考文献列表格式：

```
## 参考文献

[1] 作者. 标题. 来源/出版物, 日期. URL（如有）
[2] 机构名. 报告/文件标题. 发布日期. URL（如有）
[3] 法律法规全称. 施行日期. 条款号（如适用）
```

**示例：**

```
## 参考文献

[1] 最高人民法院. 《关于审理建筑物区分所有权纠纷案件具体应用法律若干问题的解释》. 2009年10月1日施行
[2] 王利明. 物权法研究（第四版）. 中国人民大学出版社, 2021年
[3] 中国物业管理协会. 2023年全国物业管理行业发展报告. 2023年12月
```

#### 引用标注规则

| 引用类型 | 文中标注方式 | 示例 |
|---------|------------|------|
| 数据/统计 | 上标编号 | 全国有**2.8亿**平方米的地下车库面积[1] |
| 法律法规 | 上标编号 | 根据《民法典》第274条[2]的规定 |
| 学术观点 | 上标编号 | 有学者认为业主共有权的边界需要重新界定[3] |
| 案例引用 | 上标编号 | 在某小区地下车库产权纠纷案中[4] |
| 常识性事实 | 无需标注 | 物业费是业主的法定义务 |

#### 验证注意事项

- **法条引用必须精确**：条款号、施行日期、是否现行有效，都必须核实
- **数据引用必须可溯源**：统计数字必须能找到原始发布机构
- **案例引用必须真实**：如为真实案例，需核实案号或公开报道；如为虚构案例，需明确标注"假设案例"
- **观点引用必须准确**：引用某人的观点时，需确认该观点确实出自此人，而非二手转述的误传
- **网络来源需确认链接有效**：引用的URL需确认可正常访问，死链不列入参考文献

### 步骤4：在 Markdown 中使用 WeChat 安全排版

> **⚠️ 关键约束：微信公众号草稿箱会剥离 <div> 容器样式。** 之前用的 :::block 模块（hero/verdict/callout/cta）在本地预览时正常，但推送到微信草稿后，框的背景、边框、圆角会消失，仅剩裸文字，破坏阅读体验。**绝对不要再使用 :::block 容器类模块。**

> **核心原则：内容第一，样式第二。** 排版是为了提升阅读体验。微信原生支持 ## 标题、**加粗**、> 引用、--- 分割线、- 列表——这些最稳定，优先使用。

Agent 读取用户 Markdown 后，根据内容结构**按需**添加排版。不使用任何 :::block 容器类模块，改用微信原生支持的格式：

#### 标题防重复规则（重要）

- **正文中不要写 # 文章主标题（H1）**：微信公众号草稿栏会自动显示文章标题（由 publish 命令的 --title 或 YAML frontmatter 的 title 控制），正文再写 H1 会导致标题显示两次
- **正文中不要重复出现文章标题**：文章标题已在微信标题栏展示，正文中任何位置（包括开头、章节标题、结语）都不应再次出现与文章标题相同的文字，避免读者看到重复信息
- **正文中只用 ##（H2）和 ###（H3）做分节标题**：## 第一节、## 第二节，不做标题嵌套
- **正文首个 ## H2 如果与 --title 相同，converter 会自动去掉**：防止微信标题栏和正文重复显示同一标题

#### WeChat 安全排版方案

| 需求 | 之前的不安全写法 (:::) | 替换为安全的原生 Markdown |
|------|----------------------|--------------------------|
| **文章开头** | :::hero 带 title/eyebrow/subtitle | > + 开场引语，或直接正文 |
| **强调核心观点** | :::verdict 卡片 | **💡 核心：** + 加粗文字 |
| **提示/引用** | :::callout 卡片 | > 💡 + 提示内容（微信原生 blockquote） |
| **行动号召** | :::cta 卡片 | --- 分隔线 + **📌** + 结语文字 |
| **问答列表** | :::faq 卡片 | **Q：**xxx + 换行 + **A：**xxx |

#### 排版语法速查

```
> 开场引语或副标题

**💡 核心：** 加粗突出文章的中心判断。

> 💡 提示或补充说明（替代 callout）

---

**📌 结语：** 行动号召文字（替代 cta）
```

#### 重点内容强调样式（重要）

文章中的关键信息必须用视觉标记强调，避免大段纯文字缺少层次感。微信支持的安全强调方式：

| 强调级别 | 写法 | 适用场景 |
|---------|------|---------|
| **核心观点** | `**💡 核心：** 观点内容` | 每篇文章1-2次，突出最关键判断 |
| **重点结论** | `**结论：**` 或 `**关键发现：**` | 章节末尾总结 |
| **重要数据/事实** | `**数字加粗**` | 关键数字、时间、版本号 |
| **警告/坑点** | `⚠️ **注意：**` 或 `**坑点：**` | 踩坑经验、易错提醒 |
| **对比强调** | `✅ 正确做法` / `❌ 错误做法` | 方案对比 |
| **经验总结** | `📌 **经验：**` | 实战经验提炼 |

**使用原则：**
- **每个章节至少1处加粗强调**：避免整章纯文字无视觉锚点
- **加粗占比 5-15%**：太少看不出重点，太多等于没加粗
- **emoji + 加粗组合**：💡📌⚠️✅❌ 搭配加粗，视觉区分度高
- **不要只加粗单个词**：至少加粗一个完整短语或句子，方便手机阅读

#### 排版使用原则

- **⚠️ 绝对不要使用 :::block 容器模块**（hero/verdict/callout/cta/toc/faq）——这些在微信草稿中会被剥离样式
- **⚠️ 正文不要写 # 文章主标题（H1）**——微信草稿API已经有标题字段，正文重复写会显示两次
- **⚠️ 正文不要重复出现文章标题**——标题已在微信标题栏展示，正文中任何位置都不应再次出现相同文字
- **优先原生语法**：## 标题、**加粗**、> 引用、--- 分割线、- 列表——这些微信原生支持，最稳定
- **emoji + 加粗辅助视觉**：💡📌📖⚠️✅❌ 等 emoji 搭配加粗，替代曾经用的卡片样式
- **20000 字符上限**：原生语法字符占用少，更多空间留给正文

### 步骤4.5：自动生成并插入配图

> **核心原则：图文相辅，不为配图而配图。** 配图是为了帮助读者理解复杂概念、建立视觉记忆，而不是装饰。每张图都必须有"为什么这里需要一张图"的理由。

Agent 在完成排版模块插入后，分析文章内容，**按需**生成配图并插入到 Markdown 文章的对应位置。

配图有两种方案，根据内容类型选择：

| 方案 | 适用场景 | 优点 |
|------|---------|------|
| **方案A：SVG 直出（首选）** | 结构图、流程图、数据图、层级图、概念示意图等所有可以用几何图形+标签表达的图 | 精确可控、文字清晰、文件小、公众号渲染友好、无需调用外部模型 |
| **方案B：image_gen AI生图** | 需要实际照片风格、写实场景、复杂视觉效果的图 | 视觉丰富、可处理复杂构图 |

**优先使用方案A**。大多数公众号文章的配图需求（层级结构、流程、对比、概念示意）都可以用简洁的 SVG 完成。

#### 配图触发条件

| 场景 | 配图类型 | 说明 | 示例 |
|------|---------|------|------|
| 文章解释**层级/递归/嵌套结构** | 结构示意图 | 将抽象的层级关系可视化 | 笔→线段→中枢→走势类型的递归结构 |
| 文章描述**流程/步骤/决策链** | 流程图 | 帮助读者跟随逻辑推进 | "本级别找中枢→次级别比力度→高级别定方向" |
| 文章涉及**多个维度对比** | 对比图/矩阵图 | 表格不够直观时用图 | 股票 vs 期货的级别组合对比 |
| 核心概念需要**直观理解** | 概念示意图 | 用图形帮助读者建立直觉 | "5K重叠"中枢的K线形态示意 |
| 文章有**前后对比/演变** | 前后对比图 | 展示变化过程 | 背驰前后力度对比 |

#### 不需要配图的场景

- 纯观点/评论类内容（配图反而分散注意力）
- 已有表格清晰展示数据
- 短文（< 1000字），图片会打断阅读节奏
- 概念本身足够简单，文字已能清晰传达

#### 配图数量原则

- **图文并茂是硬要求**：文章必须包含配图，纯文字文章体验差。图片管道（SVG→PNG→CDN上传）已跑通，应充分利用
- **3-5张为宜**：一篇公众号文章配3-5张配图效果最好，1张太少，5张以上打断阅读节奏
- **封面不算在内**：封面是独立的，配图指正文中的插图
- **论文风格版优先配图**：对话版一般不配图（保持对话节奏），论文版在关键概念处配图
- **长文必须多配**：超过3000字的文章，每增加1000字至少多配1张。7000-8000字的长文应有5张以上配图
- **图文比例协调**：图片间距均匀，不要集中在一个章节

#### 配图位置规则（重要）

- **每个章节开头放一张配图**：文章的每个主要章节（## H2 级别）开头都应放置一张与该章节内容相关的配图，图片紧跟在章节标题之后、正文文字之前
- **长章节灵活添加**：如果某个章节文字超过1500字，除了章节开头的配图外，可在章节中间适当位置再添加1张配图，保持图文节奏
- **配图与内容协调**：每张配图必须与所在章节的内容直接相关，不可为凑数而插入无关图片
- **图片位置示例**：

```markdown
## 物业费的法律性质

![物业费构成示意图](output/images/01_property_fee_structure.png)

物业费不是"给物业公司的服务费"，而是业主为维护共有部分而承担的法定义务。这个区分至关重要......

（如果本章节文字超过1500字，可在中间位置再插入一张配图）

![业主共有部分范围](output/images/02_common_area.png）

共有部分的界定直接决定了物业费的合理范围......
```

#### 方案A：SVG 直出配图 → 自动转为 PNG（首选）

对于结构图、流程图、层级图等可以用几何图形+文字标注表达的图，直接写 SVG 代码生成，用 `write_to_file` 工具写入 `output/images/` 目录。**注意：微信公众号不支持 SVG 格式，需要转为 PNG 后上传到微信 CDN。**

> **⚠️ 微信图片要点：**
> 1. SVG 不能直接在微信文章中显示（微信仅支持 JPG/PNG/GIF）
> 2. 本地路径图片不能直接显示，必须上传到微信 CDN
> 3. **SVG→PNG 转换优先用 Playwright 渲染**——cairosvg 无法正确渲染中文字体（微软雅黑/苹方会变成衬线字体），导致配图文字模糊。Playwright 是微软开源的浏览器自动化库（`pip install playwright && playwright install chromium`，首次需下载约 150MB Chromium 内核），本地运行无 API 费用。Agent 应先检测 Playwright 是否可用，可用则用 Playwright 转换；不可用则回退 cairosvg（仅限纯英文 SVG）或改用 image_gen 直接生成 PNG
> 4. `wechat-pub publish` 内置的 cairosvg 转换仅适用于纯英文/数字的 SVG，中文内容必须提前手动用 Playwright 转好 PNG

##### SVG 通用模板

  <style>

|------|--------|------|

##### SVG→PNG 转换（Playwright 渲染）

**优先使用 Playwright 转换**。cairosvg 无法正确渲染中文字体（微软雅黑/苹方会变成衬线字体），仅当 Playwright 不可用且回退方案可接受时使用。

转换脚本模板（Agent 应在 `output/images/` 目录下临时创建并执行，用完删除）：

```python
import os, glob, re, base64

svgs = sorted(glob.glob('0*.svg'))  # 只转换正文配图，排除封面

# 优先 Playwright（中文渲染正确），不可用则回退 cairosvg（仅限英文内容）
try:
    from playwright.sync_api import sync_playwright
    print("Using Playwright for SVG→PNG conversion...")
    with sync_playwright() as p:
        browser = p.chromium.launch()
        for svg_file in svgs:
            png_file = svg_file.replace('.svg', '.png')
            with open(svg_file, 'r', encoding='utf-8') as f:
                svg_content = f.read()
            m = re.search(r'width="(\d+)".*?height="(\d+)"', svg_content)
            w, h = int(m.group(1)), int(m.group(2))
            page = browser.new_page(viewport={'width': w, 'height': h}, device_scale_factor=2)
            svg_b64 = base64.b64encode(svg_content.encode('utf-8')).decode('utf-8')
            page.goto(f'data:image/svg+xml;base64,{svg_b64}', timeout=60000)
            page.wait_for_timeout(500)
            page.screenshot(path=png_file, timeout=60000)
            page.close()
        browser.close()
except ImportError:
    import cairosvg
    print("WARNING: Playwright not available, falling back to cairosvg (Chinese fonts may render incorrectly)...")
    for svg_file in svgs:
        png_file = svg_file.replace('.svg', '.png')
        cairosvg.svg2png(url=svg_file, write_to=png_file, scale=3)
```

转换完成后，在 Markdown 中直接引用 `.png` 路径：`![图描述](output/images/xx.png)`

### 步骤4.6：字数强制检查

> **⚠️ 微信公众号单篇文章字数上限为 20000 字符（含 HTML 标签和图片标签）。Agent 必须控制纯文本字数在 7000-8000 字之间，确保总字符（纯文本 + HTML 标签 + 配图标签 + 排版语法）落在 19000-20000 区间内。**

> **字数分配逻辑**：纯文本 7000-8000 字 + HTML/排版标签约 6000-8000 字符 + 配图（每张约 800-1200 字符的 img 标签）5-6张约 4000-6000 字符 + 参考文献约 500-1000 字符 ≈ 总计 19000-20000 字符。纯文本字数过多会导致总字符超出 20000 上限。

Agent 完成排版和配图后，必须执行字数检查：

```
1. 统计纯文本字符数（去除 HTML 标签、Markdown 语法、图片标签）
2. 统计总字符数（含 HTML 标签、图片标签、排版语法等所有字符）
3. 纯文本字数检查：
   - 如果 < 7000 字 → 返回步骤3/文章改写阶段，补充论证和案例
   - 如果 > 8000 字 → 精简内文，删减冗余论述，确保纯文本不超过 8000 字
4. 总字符数检查：
   - 如果 < 19000 字符 → 适当补充内容或增加配图
   - 如果 > 20000 字符 → 精简内文或减少排版标签，确保不超上限
5. 发布时 wechat-pub publish 会自动拦截超限文章
```

| 方法 | 示例 |
|------|------|
| 补充类比 | "这个过程就像盖楼：K线是砖块，笔是墙体，线段是楼层……" |
| 添加反例 | "如果离开段没有出现背驰，而是放量突破，这时候应该……" |
| 延伸应用 | "同样的逻辑还可以用在 XXX 市场，区别在于级别的选择……" |

## Agent 规则

1. **优先扫描 data/ 文件夹**：检查当前目录下 `data/` 文件夹是否存在，存在则按分组规则批量处理
2. **再判断输入类型**：完整文章跳过改写，AI聊天记录双输出，笔记/零散想法单输出
3. **批量逐篇处理**：多篇文章逐篇执行步骤1-7，每篇独立改写、排版、推送，最后输出汇总
4. **⚠️ 字数优化约束（关键）**：
    - 微信公众号文章内容上限为 **20000 字符**（含 HTML 标签和图片标签）
    - **纯文本字数目标：7000-8000 字** —— 纯文本过多会导致总字符超出 20000 上限
    - **总字符目标：19000-20000 字符** —— 纯文本 + HTML标签 + 配图标签 + 排版语法 + 参考文献的总和
    - **字数分配逻辑**：纯文本 7000-8000 字 + HTML/排版标签约 6000-8000 字符 + 配图 5-6张约 4000-6000 字符 + 参考文献约 500-1000 字符 ≈ 19000-20000 字符
    - **内容第一，样式第二**：在字数限制内尽可能充实内容，排版样式服务于内容
    - **排版适量不过度**：按需使用微信原生排版语法，对读者阅读体验有益就加，仅为好看就不加
    - **充分展开是硬约束，不是建议**：
      * 对原文中的每个关键论点，至少补充 **2-3 个具体例子、类比或延伸分析**
      * 每个回答段落展开到 200-400 字（中文），不要只有寥寥几句
      * 引入同类比较（如"这与 X 类似，但不同之处在于……"）
      * 添加"如果……会怎样"的假设性推演，拓宽讨论维度
      * 对专业概念做通俗解释 + 实际应用场景举例
    - **禁止过度精简**：不要为了"简洁"而删减有价值的内容
    - **⚠️ 纯文本字数上限**：纯文本不可超过 8000 字，否则加上 HTML 和配图标签后总字符会超出 20000 上限
    - **⚠️ 字数检查**：纯文本必须在 7000-8000 字之间，总字符必须在 19000-20000 之间
    - **扩充方法**：如果纯文本不足 7000 字，回到改写阶段补充论证和案例。参考步骤4.6的扩充技巧
5. **双输出处理**：AI聊天记录需要生成两个独立的 Markdown 文件，分别推送两个草稿
6. **风格区分**：对话整理版和论文风格版使用不同的改写规则和语言风格，不能互相混淆
7. **自动增强排版**：用户只需提供纯 Markdown，Agent 负责分析内容并**按需**用微信原生语法（## / > / **加粗** / --- / 列表 / emoji）增强排版。**禁止使用 :::block 容器模块**，这些在微信草稿中会被剥离样式
8. **自动生成配图（图文并茂是硬要求）**：按步骤4.5的规则，分析文章内容，生成并插入配图。**配图位置规则**：每个主要章节（## H2）开头放一张配图，紧跟章节标题之后、正文之前；如果某章节文字超过1500字，可在章节中间灵活添加1张配图。每篇文章至少5张配图。**注意 PNG 格式**：SVG 不能直接用于微信，必须提前用 Playwright 转为 PNG（cairosvg 中文渲染有字体问题，不可用），Markdown 中直接引用 PNG 路径
9. **先 inspect 再操作**：排版和配图完成后，先 inspect --json 确认无 error
10. **先 capabilities**：不确定命令是否存在时，先执行 capabilities --json
11. **排版适量，内容优先**：hero + cta 是标配，verdict/callout/toc/faq 按需使用，以"对读者有用"为标准，不堆砌样式模块
12. **草稿需要封面**：没有封面时用 --auto-cover 自动生成。封面底部显示公众号名称，用 --author "公众号名" 传入
13. **摘要自动生成**：不需要手动指定 digest，微信会自动截取
14. **JSON 输出**：Agent 调用 CLI 时始终加 --json，便于解析（注意 publish 命令无 --json，以纯文本输出）
15. **确认后再推送**：先 preview 让用户确认效果，再 publish
16. **微信凭证安全**：配置文件在 ~/.config/wechat-pub/config.yaml，不在项目目录内，不会随代码提交
17. **避免 AI 味**：严格遵循语言风格规范中的禁用表达列表
18. **文件存放**：生成的改写稿统一保存在 `output/` 文件夹下，以 `{话题名}_对话版.md` 和 `{话题名}_论文版.md` 命名。`output/` 不纳入 Git 仓库
19. **data/ 文件夹不提交**：`data/` 是临时输入目录，不应出现在 git 仓库中

## 关于封面

- `generate-cover` 生成 Claude 风格封面（标题+几何装饰+主题色）
- 如需 AI 配图，Agent 可用 image_gen 工具生成，然后 `publish --cover 图片.png` 指定
- 封面底部会显示公众号名称，通过 `--author` 参数传入，例如 `--author "张三的交易笔记"`
- 封面文件保存在 `output/images/cover.png`（自动生成时）

## 微信公众号配置

1. 登录 [微信公众平台](https://mp.weixin.qq.com)
2. 进入「开发 → 基本配置」→ 获取 AppID 和 AppSecret
3. 将服务器 IP 加入白名单（或使用测试号）
4. 运行 `wechat-pub config init`，配置文件在 `~/.config/wechat-pub/config.yaml`
5. 或设置环境变量：`WECHAT_APP_ID` / `WECHAT_APP_SECRET`

**安全说明**：配置文件存储在用户主目录 `~/.config/wechat-pub/` 下，不在项目目录内，推送到 GitHub 不会暴露凭证。

**测试号**：如需测试，可在「开发 → 开发者工具」申请测试号，不受 IP 白名单限制。

## 我为什么想写公众号

微信公众号是个很好的平台。它没有字数限制，不像微博那样碎片化；它的读者是主动订阅的，不像信息流那样稍纵即逝；它的排版空间足够，不像朋友圈那样只能发几张图。

我一直觉得，在公众号上写长文是一种从容的表达方式——有空间把一件事说透，有读者愿意花时间读。

但每次坐下来想写点什么，都会被同一件事卡住：**排版**。

## 排版这件事，劝退了我半年

我注册公众号有半年了。第一篇文章写了两个小时，排版花了一个小时。当时想，下次会快的。

第二篇，排版还是一小时。第三篇，依然一小时。

不是排版有多难，而是**太碎了**。每次都是同样的流程：

- 在 Markdown 里写完文章
- 打开 135Editor，选模板，拖组件
- 复制到微信编辑器，格式变了
- 调字号、调间距、调颜色
- 找封面图，裁成 2.35:1
- 写摘要，新建图文，上传封面，保存草稿

**每一步都不难，但每一步都要人肉操作。** 写 Markdown 的人最烦的就是离开键盘去拖拽。更烦的是，这些操作每次都重复——你上周排好的版，这周还得重新来。

:::callout
type: tip
body: 痛苦的根源只有一个：从写完文章到发布之间，没有一个自动化的管道。每个环节都需要手动介入，消耗的不仅是时间，更是写东西的热情。
:::

## 发现 md2wechat：思路对了，但不够

后来我在 GitHub 上发现了 **md2wechat**——一个用 Markdown 写公众号、自动排版的工具。

它的思路跟我一模一样。而且做得很用心：43 个排版模块、40 多套主题、Go 语言写的 CLI，还适配了 CodeBuddy、Claude Code 等 AI IDE。

我兴奋地试了一下，然后发现一个关键问题：**排版引擎是闭源付费 API**。你的文章内容会发到作者的服务器上处理，按调用量付费。

不是说这样设计不对——作者完全有权这么做。但对用户来说，这意味着你的文章内容经过第三方服务器，工具的可用性依赖别人的服务，而且想自定义排版细节也做不到。

:::verdict
eyebrow: 核心判断
title: 我需要的不是更好的工具，而是自己能掌控的工具
body: 现有方案要么不自动化，要么不自主。我需要的工具必须同时满足两个条件：全自动 + 完全自托管。
:::

## 用本质探索方法论拆解问题

面对"自己造一个工具"这个目标，我用了一个思维框架来拆解：**本质探索方法论**。

这是我之前整理的一个开源项目：[essence-programming](https://github.com/zh2673-git/essence-programming)。核心很简单，三个递进的问题：

1. **是什么**——这个问题到底是什么？
2. **为什么**——为什么现有方案不够好？
3. **怎么做**——该怎么做？

### 是什么：精确定义问题

"排版太烦"是个模糊的抱怨。精确地说，**公众号排版的本质是：把 Markdown 转换为微信兼容的 HTML**。

关键词是"微信兼容"。微信编辑器有大量限制：删除 `<style>` 标签、忽略 `class` 属性、不支持外部 CSS、20000 字符限制……所以普通的 Markdown 转 HTML 工具在微信里会变成没格式的纯文本。不是工具不好，是微信的渲染环境太特殊。

### 为什么：分析方案不足

现有方案要么不自动化（微信编辑器、135Editor），要么不自主（md2wechat、mdnice 的排版引擎是闭源付费 API，文章内容经第三方服务器）。自动化和自主性不可兼得——除非你自己实现排版引擎。那就自己造一个。

### 怎么做：拆解再重构

我拆解了 md2wechat 的架构，然后用自己的方案重新实现：

- 它用 Go，我用 Python——开发快，AI IDE 生态好
- 它有 43 个模块，我只做了 6 个核心模块——80% 的文章只用 6 种排版结构
- 它没做微信推送，我做了——这才是"一键发布"的关键
- 它没做 AI 配图，我做了占位封面——先跑通再优化

6 个模块不是偷懒，是克制：

- **hero**——开头大标题，3 秒抓住读者
- **callout**——提示框，关键信息跳出来
- **verdict**——核心判断，一篇文章只有一个主论点
- **toc**——目录，长文必备
- **faq**——问答折叠，节省篇幅
- **cta**——末尾行动号召

一篇文章不超过 5 个模块，开头放 hero，末尾放 cta，中间按需放 callout 和 verdict。不用做选择，结构自然清晰。

## 这个工具的特色

和 md2wechat 相比，md2wechat-py 有几个自己的特色，都是在解决实际问题时逼出来的。

### 特色一：纯内联样式

微信编辑器会**删掉 `<style>` 标签、忽略 `class` 属性**，所以所有 CSS 类方案在微信里都失效。我踩过这个坑——本地预览很漂亮，推到微信一看纯文本白纸一张。

解决方案是 `InlineStyler`：用正则匹配所有没有 `style` 属性的裸标签（`<h2>`、`<p>`、`<blockquote>` 等），根据主题自动注入内联样式，同时跳过排版模块模板里已有样式的元素。每个 HTML 元素都自带完整样式，推到微信里渲染效果跟本地预览完全一致。

### 特色二：6 个排版模块 + `:::block` 语法

```markdown
:::hero
eyebrow: 工具诞生记
title: 拆解一个工具，造了自己的
subtitle: 专注写内容，排版的事交给工具
:::
```

处理流程：`LayoutRenderer` 正则匹配 `:::block name ... :::` → 解析 YAML 字段 → 找到对应 Jinja2 模板注入字段 → 替换回原文 → markdown-it 渲染剩余 Markdown。

每个模块定义是分层的：`layout/opening/hero.yaml` 定义字段，`templates/opening/hero.html` 定义模板。新增模块只需加这两个文件，重启 CLI 自动发现。

### 特色三：3 套 Claude 风格主题

参考 Anthropic 的 Claude 品牌设计语言，提炼出三套风格：`claude-warm`（赭石橙 + 奶油白，默认）、`claude-clean`（白底 + 赭石点缀）、`claude-dark`（深色暖棕调）。

每套主题不只是换颜色——字号体系（h1 22px / h2 19px / h3 17px）、行高 1.8、圆角、间距都是整体调过的，确保视觉节奏一致。

### 特色四：一键推送到微信草稿箱

md2wechat 没做的功能。调用微信素材上传 API 上传封面图，再调用草稿箱新建 API 创建草稿。access_token 缓存、素材 media_id 引用、封面 2.35:1 比例要求都封装进了 `wechat/client.py`。

对用户来说就是：`wechat-pub publish article.md --auto-cover`

### 特色五：Agent 原生设计

所有 CLI 命令支持 `--json` 输出，统一的 JSON 格式方便 Agent 直接解析。我在 CodeBuddy 上写了 SKILL.md，定义了完整执行流程——用户只需要说"发布到公众号"，剩下全自动。

## 踩过的坑

上面的一些技术方案，背后都是踩坑踩出来的。

**坑1：微信删 `<style>` 标签**——本地预览正常，推到微信格式全丢。排查后发现微信删 `<style>` 标签、忽略 `class`。最终写 `InlineStyler` 注入内联样式。

**坑2：`:::block` 替换后 Markdown 渲染断裂**——替换后的 HTML 和后续 Markdown 之间没有空行，markdown-it 把标题当 HTML 块的一部分输出纯文本。修复：替换时加空行。

**坑3：20000 字符限制**——内联样式比 CSS 类长很多，一篇普通文章就超标。修复：HTML 压缩——移除注释、压缩多行 style 为单行、去除多余空白。

## 最终效果

踩完坑之后，整个发布流程变成了：

**写完 Markdown，一条命令发布：**

```bash
wechat-pub publish article.md --auto-cover --json
```

打开微信公众号后台，草稿箱里就躺着排版好的文章。从写完到发布，30 秒。

配合 AI IDE（比如 CodeBuddy），更简单——把文章内容发给 Agent，说"发布到公众号"，Agent 自动完成：分析内容、插入排版模块、生成封面、推送草稿箱。

## 这才是我想要的写作方式

有了这个工具，我终于可以专注在内容上了。

不用纠结排版——写 Markdown 就好，AI 自动增强排版。不用纠结格式——Claude 风格的主题已经调好了，赭石橙配奶油白，温暖又专业。不用纠结发布流程——一条命令，从本地到草稿箱。

**我只需要做一件事：把想法写出来。**

接下来的日子，我会在这里分享更多内容：AI 工具和实践、法律知识和思考、生活中有意思的想法……不拘泥于某个领域，想到什么写什么。

如果你也觉得公众号排版是个痛点，也想用 Markdown 写公众号，欢迎试试这个工具。

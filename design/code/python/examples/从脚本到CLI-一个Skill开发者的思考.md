我最近在做一个微信公众号的 Skill，从开发过程中对"脚本"和"CLI 工具"的区别有了更深的理解。这篇文章记录一下这个思考过程。

## 为什么 Skill 要用 CLI 而不是脚本

以前写 Skill，逻辑都塞在脚本里——10 个函数写在一个文件里，想调用某个功能就得改入口代码。脚本跑起来简单，但大模型想调用它，就麻烦了。

CLI 工具不一样。比如我开发的 `wechat-pub`，它有 inspect、convert、publish 这些子命令，每个命令有明确的参数格式。大模型只需要知道"先 inspect 检查、再 publish 推送"这个流程，生成对应的命令行就行，不用管代码内部怎么组织的。

:::callout
type: tip
body: 用一个类比：脚本像一把瑞士军刀，所有功能叠在一起；CLI 像一个工具箱，每个工具有自己的位置和用法。大模型用工具箱比用瑞士军刀顺手得多。
:::

## CLI 的容错：不是自动纠错，是明确反馈

有人问 CLI 容错率高不高。这个问题要看怎么理解。

CLI 本身对语法很敏感，少个空格、参数顺序错了都会直接报错。但它有一个优势：**反馈很明确**。成功了返回 0，失败了返回非 0，还会在 stderr 里告诉你具体哪里错了。大模型拿到这个错误信息，就能调整参数重试。

相比之下，传统脚本如果没写完善的错误处理，可能跑失败了都没个像样的提示。模型不知道哪里出了问题，也没法自我纠正。

所以 CLI 的"容错"更准确地说是"让模型有能力纠错"，而不是自己帮你纠错。

## Skill.md 的职责：写流程，不写逻辑

有了 CLI 工具后，Skill.md 该写什么？

只需要两部分：

1. **YAML 元数据**：告诉大模型这个技能叫什么、什么时候该用
2. **执行流程**：第一步调哪个命令、传什么参数，第二步怎么处理返回结果

具体的业务逻辑——比如标题样式怎么套、HTML 怎么生成——都封装在 CLI 工具内部。Skill.md 只负责"流程编排"，不负责"逻辑实现"。

这样做的好处是：改功能只需要更新 CLI 工具，不用动 Skill.md。比如我后来优化了封面生成的模板，Skill.md 完全不用改，因为流程没变，只是 CLI 内部的实现更精致了。

## 我的完整工作流

回到我做公众号 Skill 的场景，整个流程现在是这样的：

1. 语音聊天产生素材（比较零散、口语化）
2. 大模型把素材整理成结构化的 Markdown 文章
3. 按公众号风格改写语言（去口语化、调整语气）
4. 调用 CLI 工具转换为 Claude 风格的 HTML 排版
5. 自动推送到微信公众号草稿箱

第 2 和第 3 步是关键——不只是格式转换，而是从"原始素材"到"可发布内容"的完整链路。这也是 Skill 演进的方向：从单纯的工具调用，变成"改写 + 排版 + 推送"的全流程自动化。

:::verdict
eyebrow: 核心观点
title: 开发 Skill 的本质是把"人怎么做事"抽象成"模型怎么调用工具"
body: 脚本时代是人告诉脚本做什么，CLI 时代是模型通过标准化接口调用工具。这个转变看似只是技术选择，实际上影响的是整个工作流的设计思路——你得从"大模型能不能理解和调用"的角度去设计每一个环节。
:::

这个公众号 Skill 的开发过程，就是这样一个不断调整视角的过程。
